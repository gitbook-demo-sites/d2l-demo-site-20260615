# Release communication model

The release-note subscription article shows a useful behavior pattern: monthly release notes live in one predictable place, and subscribers opt in to updates from a single discussion thread.

## What the current source pattern already does well

- Keeps release updates anchored to one recurring discussion
- Lets customers choose bookmarked-post notifications instead of broad email noise
- Gives D2L a consistent monthly communication ritual

## How to present it more clearly in the hub

{% tabs %}
{% tab title="For customers" %}

Explain the outcome first:

- Subscribe once
- Get notified when a new monthly release lands
- Adjust notification mode to popup, email, or both

Then link out to the specific discussion thread and the latest release family.
{% endtab %}

{% tab title="For D2L teams" %}

Use the page as a canonical support answer for:

- "How do I hear about product changes?"
- "Where are the latest Brightspace release notes?"
- "Why did I stop receiving notifications?"

This cuts repetitive support handling and creates one URL worth sharing.
{% endtab %}
{% endtabs %}

## Suggested page pattern

1. Start with a short explainer of what users get by subscribing
2. Show the three steps to subscribe to the release-updates discussion
3. Explain notification preferences separately so users can reduce noise
4. Link to the latest release-note collection and roadmap surfaces

{% hint style="success" %}
For a live demo, this page is a strong way to show GitBook as a product-communication layer, not only a static knowledge base.
{% endhint %}
