# Docs operating model

This demo uses one site with distinct section ownership rather than one giant mixed navigation tree.

## Proposed ownership

<table>
  <thead>
    <tr>
      <th>Section</th>
      <th>Primary owner</th>
      <th>Content type</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Home</td>
      <td>Docs program or product marketing</td>
      <td>Navigation, positioning, audience routing</td>
    </tr>
    <tr>
      <td>Help center</td>
      <td>Support enablement</td>
      <td>Task guidance, escalation paths, release-update workflows</td>
    </tr>
    <tr>
      <td>Data sets API reference</td>
      <td>Developer platform team</td>
      <td>Spec-backed reference and models</td>
    </tr>
  </tbody>
</table>

## Why this is healthier than a single mixed tree

- Support pages can stay concise and task-driven
- API pages can keep spec truth without prose drift
- Homepage and section landing pages can absorb product storytelling
- Different teams can own different sections without fighting one nav model

## Good follow-up expansion areas

- Instructor and admin help journeys
- Status and release calendar pages
- A role-aware landing flow using adaptive content
- Shared reusable blocks for support-contact language and account-ownership disclaimers
